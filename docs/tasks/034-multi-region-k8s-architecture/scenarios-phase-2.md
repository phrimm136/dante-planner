## Plan-time decisions (mine, fixed once, BINDING for this phase's spawns)
- **D1 — Read-path seam mechanism (§9.4 PLAN-TIME):** a single concrete `@Component ByIdReadGuard` in `org.danteplanner.backend.shared.readpath`, method `<T> T read(String entityType, UUID id, java.util.function.Supplier<T> dereference)`; Phase-2 body = `return dereference.get();` (pass-through). Wired at the **controller** level (outside `@Transactional`, so Phase 8 GTID gate / Phase 3 re-check can steer routing before the tx opens), wrapping the service byId call in the supplier. Precedent for controller-level infra collaborators: `PlannerQueryController` already calls `rateLimitConfig.checkCrudLimit(...)` and `ClientIpResolver.resolve(...)`. Chosen over AOP (hidden, CLAUDE.md prefers explicit DI) and repository decorator (can't reach the request cookie for Phase 8, can't re-run under forced-primary for Phase 3, sits inside the tx). One concrete class, no interface (YAGNI — one impl that evolves, no substitution need). The entityType literal `"planner"` was extracted to `ByIdReadGuard.PLANNER_ENTITY_TYPE` in the refactor leg.
- **D2 — Routing datasource:** `RoutingDataSourceConfig` in `shared/config`, `@Configuration` + `@ConditionalOnProperty(name="datasource.routing.enabled", havingValue="true")` + `@EnableConfigurationProperties({DataSourceProperties, ReplicaDataSourceProperties})`, exposes `@Bean @Primary DataSource` = `LazyConnectionDataSourceProxy` over an `AbstractRoutingDataSource` (`ReadOnlyRoutingDataSource`) keyed on `TransactionSynchronizationManager.isCurrentTransactionReadOnly()` → REPLICA when read-only AND replica enabled, else PRIMARY. Oregon (replica disabled) → both keys map to the single PRIMARY pool. URLs/creds from properties (`spring.datasource.*` primary via Boot `DataSourceProperties`, `datasource.replica.*` replica via `@ConfigurationProperties(prefix="datasource.replica")`); **pool sizes from PoolLedger constants, never literals**. When routing disabled (default) the bean is absent and Spring Boot's autoconfigured datasource stands → existing tests unperturbed (surgical precision, CLAUDE.md rule #1). Proven live by RoutingSeoulIT (scenario 6). This is the "same image, different config" profile/property-driven model the Considerations require.
- **D3 — PoolLedger constants (§6 BINDING):** class in `shared/config`. `OREGON_PRIMARY_POOL=15`, `SEOUL_PRIMARY_POOL=10`, `SEOUL_REPLICA_POOL=15`, `MAX_PODS_PER_ASG=2`, `RDS_MICRO_MAX_CONNECTIONS=85`, `CONNECTION_RESERVE=10`. Re-check bulkhead pool is Phase 3 (not added here). INV9 invariant: `(OREGON_PRIMARY_POOL + SEOUL_PRIMARY_POOL) * MAX_PODS_PER_ASG <= RDS_MICRO_MAX_CONNECTIONS - CONNECTION_RESERVE` → 50 <= 75 (only pools that hit the shared Oregon primary count; the Seoul replica pool hits the separate Seoul replica RDS). The datasource config reads these same constants — not copies (scenario 3 asserts production sizing from the ledger).
- **maximumPoolSize=50 guard:** not present in the repo today; must never be introduced. Prod pool sizing comes from PoolLedger via D2.
